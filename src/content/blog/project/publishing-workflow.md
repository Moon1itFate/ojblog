这个站点的文章发布会保持一个简单原则：文章由我在仓库里维护，读者通过评论区交流。

这样做的好处是边界很清晰。文章内容是长期沉淀，评论是围绕文章的讨论；一个负责表达，一个负责互动。

## 为什么不开放公共发文

个人博客最重要的是“声音的一致性”。

如果把公共发文入口直接开放出去，站点很快会变成社区系统，需要登录、权限、审核、反垃圾、内容治理和数据库维护。那当然也能做，但它会把项目的重心从个人博客拉走。

现在更适合的设计是：

- 我负责发布正式文章；
- 读者在文章底部评论；
- 有价值的讨论可以被整理进后续文章；
- OJ Tracker 的复盘结果可以逐步沉淀成项目日志或题解。

## 当前发布方式

文章文件放在 `src/content/blog` 中，按主题拆分目录。

- `algorithm`: 算法题解。
- `project`: 项目日志。
- `review`: 周复盘。
- `life`: 随笔。

每篇文章都带有 frontmatter，用来声明标题、摘要、发布时间、封面、标签和分类。

## 发布前做什么

每次新增文章后，我会先做两类检查：

```bash
corepack pnpm check
corepack pnpm build
```

前者确认类型和 Astro 页面没有错误，后者确认整站可以完整生成。通过之后再提交并推送。

## 下一步

后续会继续完善文章发布体验。

- 给常见文章类型准备模板。
- 让本地 CMS 更适合单人写作。
- 把 OJ Tracker 生成的周报转成可编辑草稿。
- 评论区启用后，建立一套简洁的互动规则。

博客不需要急着变复杂。先让写作持续发生，系统就会自然长出它需要的形状。
